## 3. Schema changes

Applied to production (`ijtgwmqrxminwypracei`) as Supabase migration **`meeting3_foundation`**, and recorded in the repo as **`supabase/migrations/0001_meeting3_foundation.sql`** (additive & idempotent):

- **`spaces`** — `id`, `project_id` (FK → `projects(id)` **on delete cascade**), `name`, `sqm` `numeric(10,2)` `default 0 check (sqm >= 0)`, `sort_order`, `created_at`. RLS **disabled** to match the sibling data tables — this is the same accepted ADR-0002 debt and must get a per-row policy in Security Phase 2 (now tracked in SECURITY.md).
- **`milestones.start_date` / `milestones.end_date`** — two **nullable** `date` columns (existing rows stay valid; the admin form enforces both on new milestones; `due_date` is set = `end_date` for backward-compatible displays).
- **Realtime** — `spaces` added to the `supabase_realtime` publication (new tables are not auto-added), so the hub's spaces live-sync actually fires.

This is the **first file-based migration**. A `0000_baseline.sql` snapshot of the hand-built live schema is still **TODO** — generate it via `supabase db dump`, do not hand-write it (see `supabase/migrations/README.md`).

## 4. Architectural decisions

- **Meeting 3 is a richer process than the current 7-stage app** (it adds staff roles, an internal-vs-client quote split, sqm pricing, milestone dates, and delivery rules). Increment 1 builds only the **role-independent, schema-light foundation**; the rest is sequenced behind the owner's open decisions.
- **Adopt the `migrations/` directory** (advances ADR-0009 from Proposed toward Accepted): every schema change is now an ordered, committed `.sql` file applied through a migration step, not hand-typed only.
- **`due_date` kept in sync with `end_date`** on new milestones so the client portal (which still reads `due_date`) keeps rendering without a forced change this increment.
