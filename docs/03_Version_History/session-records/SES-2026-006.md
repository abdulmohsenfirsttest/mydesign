## 1. Summary

The owner asked to be able to **make accounts and control what each one can see** ahead of a website-review meeting. Shipped **owner-configurable per-account permissions**.

## 2. Versions shipped

**v4.2.0 · Owner-configurable per-account permissions**
What: `admins.permissions` (jsonb) — a per-account list of admin **areas** (Projects, Project Hub, Pricing, Clients, Bookings, Quotes, Upload Files, Staff) the account may see. `NULL` falls back to the role's defaults (manager = all; designer = projects/hub/uploads; PM = projects/hub/clients/bookings). The **sidebar** and the **Pricing** and **Staff** pages gate on the effective areas (`canSee` in `lib/roles.ts`); login stores the permissions. The Staff page became **Staff & Permissions**: create an account with per-area toggles, and a **per-row editor** to change any account's role + access. The **owner row (`0547080147`) is protected** — always full access, not editable/deletable, so the owner can't lock themselves out.
Why: let the owner run access control directly (for the review meeting and beyond).
Schema: migration `0004_account_permissions` (`admins.permissions jsonb`).
Decision: MINOR. Enforcement is client-side (localStorage), consistent with ADR-0001/0010 — a person could still hit the API directly until Security Phase 2; the UI access control is real, the security boundary is not (yet).

## 3. Schema changes

Migration `0004_account_permissions` (additive), applied to prod: `admins.permissions jsonb` (nullable; NULL = role defaults).
