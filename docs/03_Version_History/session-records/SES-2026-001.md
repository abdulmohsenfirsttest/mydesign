## 1. Summary
This is the first recorded session for MyDesign and it adopts the versioning + session-record discipline retroactively across the whole build arc to date. It covers the project from the initial single-page portfolio (v1.0.0), through the black/serif "Design & Build" rebrand that matched mydesign.sa (v1.1.0), into the client-portal and owner-admin platform that turned the marketing site into a SaaS surface (v2.x), then the migration off mock/localStorage data onto a real Supabase Postgres backend (v3.0.0), and finally the quotes / milestones / files / Supabase-Realtime expansion plus phone-based admin auth (v3.2.0). Alongside the build history, this session established the supporting discipline itself: a documentation set, a code-health audit, a CI gate (warn-before-enforce), and a Drive mirror — so that from here on every change is briefed, versioned, verified against live, and recorded.

## 2. Versions shipped

**v1.0.0  ·  Initial portfolio**
What: Single-page portfolio — Hero, Projects, About, Contact — in a dark-neutral / indigo palette.
Why: Stand up a first public presence for the firm.
Schema: None (static site, no backend).
Decision: Ship a single marketing page first; defer any data layer.

**v1.1.0  ·  Black/serif "Design & Build" rebrand**
What: Full visual redesign to a pure black/white luxury aesthetic (Playfair Display headings + Inter body), fixed navbar, and a dedicated `/projects` page.
Why: Match the look and positioning of mydesign.sa; the indigo portfolio palette did not read as a high-end design-&-build firm.
Schema: None.
Decision: Lock the black/serif system as the permanent brand; the v1.0.0 palette is retired, not themed.

**v1.1.1  ·  Real photography**
What: Iterative image work — reference screenshots in, a rendering fix (`next/image` `fill` not showing, switched to plain `<img>`), then replaced with Unsplash stock photography.
Why: Stock-quality imagery to carry the luxury aesthetic; the screenshot pass exposed an image-rendering bug.
Schema: None.
Decision: Use plain `<img>` for these hero/gallery images rather than fight `next/image` `fill` layout; treat the Unsplash set as placeholder until real project photos arrive.

**v2.0.0  ·  Client portal platform (NEW ERA)**
What: Added a full client portal — custom auth, a `/book` consultation flow, and a `/dashboard` with projects, messages, files, financials, and reviews. The marketing site became one surface of a product.
Why: Move from a brochure site to a platform clients log into to follow their project.
Schema: Introduced the first data model (bookings, clients, projects and the dashboard sub-areas), initially mock/localStorage-backed.
Decision: New product surface = MAJOR. Build the portal as its own area rather than bolting onto the marketing pages.

**v2.1.0  ·  Owner/admin dashboard**
What: Added the admin/owner side — clients, projects, bookings, messages, quotes, and uploads management views.
Why: The owner needs to run the business behind the client portal (manage who is in, what they see, and what is booked).
Schema: Admin reads/writes the same client-portal entities; no separate data model yet.
Decision: One codebase, two audiences — keep `/admin/*` distinct from `/dashboard/*` with its own layout and guard.

**v2.2.0  ·  Portal workflow model**
What: Message file attachments; phone + OTP login replacing email/password; the 7-stage project workflow (Quotation → Mood Board → 2D → 3D → Plans → Payment → Delivery); and a meeting-log + milestone system with client approval that replaced the earlier mock chat UI.
Why: Model how the firm actually works — staged design delivery, logged meetings, and client sign-off — rather than a generic chat.
Schema: Stage/progress on projects; meetings (with decisions and attached files) and milestones added.
Decision: The relationship is a meeting log with explicit client approval, not a chat thread; the 7 stages are the canonical project lifecycle.

**v2.2.1  ·  Login stabilization**
What: Reverted an accidental full-portal deletion; added a "Client Login" nav link; wired the booking flow into login; and simplified auth to phone + a fixed password (`123123`).
Why: Recover from the accidental wipe and make the booking → login path actually work; the OTP flow was more friction than the use case needed.
Schema: None.
Decision: Drop OTP in favor of phone + a known default password — chosen for simplicity over security as a deliberate (and now documented) trade-off.

**v3.0.0  ·  Supabase backend (NEW ERA)**
What: Replaced all mock/localStorage data with a real Supabase Postgres backend across the client portal and every admin page; cleared the fake seed meeting data.
Why: A real, shared, persistent database is required for the portal to be usable by actual clients and the owner at the same time.
Schema: The data now lives in Postgres tables (bookings, clients, projects, meetings) via a single shared browser client (`lib/supabase.ts`) on the public anon key. (The milestones, quotes, files, and admins tables came later, in v3.2.0.)
Decision: Incompatible data model = MAJOR. All access is client-side against the anon key for now; no server-side keys.

**v3.1.0  ·  Admin operations & auth**
What: Booking status dropdown with live update; an admin client-creation form; bookings that auto-register a client; and an admin login with `/admin/*` route protection and sign-out.
Why: Give the owner real operational control and gate the admin area behind a login.
Schema: Admin login validated against an admins concept (phone + password); booking write now also upserts a client row.
Decision: A booking auto-creates a client (upsert on phone, default password) so a client can log in immediately after booking; `/admin/*` is guarded by an `admin_session` localStorage flag.

**v3.1.1  ·  Login & bookings fixes**
What: Fixed client-login phone matching and added a "+ Add to Clients" action on the bookings page. This is the last commit currently in git.
Why: Phone matching was failing on login; the owner needed a one-click way to convert a booking into a client.
Schema: None.
Decision: Keep phone as the client identity key (UNIQUE on `clients.phone`).

**v3.2.0  ·  Quotes, milestones, files & realtime expansion**
What: A quotes table + builder + a `quotes` storage bucket; per-project milestones; a dedicated files page + a `files` storage bucket; a meeting "hub" with file upload/download, client comments, and approve/unapprove; Supabase Realtime live-sync across all admin pages; an `admins` table for phone-based admin login; and a local JSON accounts backup. ~1,800 line changes across 19 files plus `supabase/admins.sql`.
Why: Round out the platform into a working operations tool — quoting, deliverables, document exchange, and live two-way sync between owner and client.
Schema: New `quotes`, `milestones`, `files`, and `admins` tables; new `files`, `meetings`, and `quotes` storage buckets; the hardcoded admin password moved out of code into the `admins` table.
Decision: Deployed to production via `vercel --prod` but NOT yet committed to git — status is in progress / pending commit. Admin auth is now table-backed (phone + password) instead of the old hardcoded `admin2026`.

## 3. Schema changes
Eight tables are referenced in code and back the live product:
- **bookings** — consultation requests (name, phone, email, service, date, time, status).
- **clients** — portal users (name, phone UNIQUE, email, password). Password stored in PLAINTEXT, default `123123`.
- **admins** — admin login (id, name, phone UNIQUE, password PLAINTEXT, created_at); seeded owner `Owner` / `0547080147`. RLS is enabled but the policy is `admins_select for select using (true)`, so the anon key can read admin passwords.
- **projects** — client_id, name, type, stage, progress, started, budget, status.
- **milestones** — project_id, sort_order, and per-milestone fields.
- **meetings** — project_id, title, date, time, type, summary, decisions (jsonb), files (jsonb), status, client_comment, approved_at.
- **quotes** — project/client link, line items, status, attached file.
- **files** — project/client document metadata.

Three PUBLIC storage buckets back uploads: **files**, **meetings**, **quotes**.

Honest provenance note: almost all of the DDL above was hand-typed directly into the Supabase SQL editor, NOT applied through a migration tool. There is **no `migrations/` directory**. The only schema artifact tracked in the repo is `supabase/admins.sql` (and even that is currently untracked / pending commit). As a result the schema is **not reproducible from the repo** — this is recorded here as a known gap to close (see §7). RLS is DISABLED on bookings, clients, projects, milestones, meetings, quotes, and files; admins has RLS enabled but with the permissive `using (true)` select policy above.

## 4. Architectural decisions
- **Custom auth, not Supabase Auth** — login compares phone + password against `clients` / `admins` directly; client default password `123123`, admin session = `localStorage.admin_session`.
- **Client-side anon access with RLS off** — all reads/writes go through one shared browser client on the public anon key; RLS being disabled on the data tables is accepted as known debt, not a feature.
- **Black/serif design system is permanent** — pure black/white, Playfair (headings) + Inter (body); the v1.0.0 indigo palette is retired.
- **7-stage project workflow is canonical** — Quotation → Mood Board → 2D → 3D → Plans → Payment → Delivery, shown on the dashboard, project detail, and admin projects.
- **Relationship model = meeting log + milestones + approval** — the owner logs meetings (with decisions and files); the client sees a timeline and can Approve/Unapprove and comment. This replaced the mock chat.
- **Storage buckets are public** — files, meetings, and quotes buckets are reachable by URL (accepted debt, see §7).
- **Realtime admin sync** — Supabase Realtime live-syncs data across all admin pages so owner edits appear without reload.
- **Booking auto-creates a client** — a `/book` submission upserts a `clients` row (on phone, default password) so the client can log in immediately afterward.
